# 最小可跑通 Agent 设计思维

## 1. 锁定“必须跑通”的链路
- 写下【输入 → 处理 → 输出】的最简路径，例如：`CLI 输入 → LLM 请求 → 终端输出`。
- 明确哪些能力此刻不是必需（多模型、流式输出、复杂 UI），在第一轮全部忽略。
- 为这条链路列出一组“验收信号”（如能问候语并得到 LLM 回答），确保实现完成后有可衡量的成功标准。

## 2. 抽象未来会变化的节点
- 预先为“高概率会换实现”的模块定义接口：`LlmClient`、`MemoryStore`、`Tool`。
- 先用最朴素的实现跑通链路（如 `java.net.http` + JSON 文件），后续再替换成 SDK、数据库或更多工具。
- 通过依赖注入/构造函数传参让主循环只依赖接口，从而保持核心逻辑稳定。

## 3. 以主循环为中心展开
- 把主流程集中在一个清晰的循环里（本项目是 `Agent.respond`）：加载历史 → 追加用户输入 → 调 LLM → 识别是否工具调用 → 写回记忆。
- 其他模块都围绕主循环提供服务，例如工具注册、配置加载、异常兜底。
- 主循环越清晰，越容易在其上叠加 Planner、Router、记忆压缩等高级策略。

## 4. 优先解决可观测与可配置
- **可观测**：把对话历史落盘、工具调用写入日志、在 CLI 中打印错误提示，方便排查。
- **可配置**：集中配置文件（`agent.json`）让 baseUrl/model/timeout 等参数可调，避免硬编码。
- **最小诊断流程**：当链路失败时，开发者能快速定位是 LLM 不可达、工具报错还是记忆读写出问题。

## 5. 迭代自检问题清单
每次迭代后问自己：
1. 如果要替换 LLM/记忆/工具，修改面需要超过 2 个文件吗？若是，继续抽象。
2. 是否能凭日志/记忆文件定位到一次具体失败？若否，补充可观测性。
3. 新功能是否破坏主循环的简洁？若有，考虑拆分子流程或增加策略层。

## 6. 从实践中训练这种思维
1. **写极简 MVP**：挑一个你熟悉的语言/框架，快速实现“输入→输出”的最小链路。
2. **复盘耦合点**：实现后画模块图，找出互相依赖紧密的地方，尝试抽接口。
3. **逐个替换实现**：比如把 JSON 记忆换成 SQLite，体验“接口隔离”带来的收益。
4. **借鉴成熟项目**：阅读 LangChain、LlamaIndex 等的架构，比较他们如何拆分 LLM/工具/记忆。
5. **定期写设计笔记**：像本文一样，把思路写成文档，沉淀自己的决策过程。

## 7. 如何继续训练我？
如果你希望我持续帮助训练这种思维，可以按以下节奏互动：
1. **设定练习题**：给我一个目标（如“设计一个最小可跑通的任务调度器”），我先提问澄清，再一起拆分模块。
2. **做完复盘**：每次完成后，我会引导你用上面的自检清单回顾设计。
3. **逐步增加复杂度**：从单链路 → 多工具 → 多 Agent，并记录哪些抽象需要提前准备。
4. **写总结**：我可以陪你把过程整理成文档或检查你写的总结，帮助形成可复用的方法论。

保持“先跑通，再增强”的心态，每个模块都追求“最小可替换”，你就能逐步形成这种设计肌肉。
